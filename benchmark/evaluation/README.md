# GraphQL Semantic Search & Introspection — Evaluation Suite

This repo is an **evaluation suite for semantic search and semantic introspection** over large
GraphQL schemas. It exists to answer one recurring question with numbers instead of opinions:
**which technique actually works best?** — which embedding model, which per-field / per-type
embedding template, which retrieval/selection algorithm, which agent, which prompt.

**The problem it's built around.** A real GraphQL schema is enormous (GitHub's is ~1.5 MB of SDL).
You can't paste it into a prompt, so to use it with an LLM you have to _semantically_ find the part
that matters — either by retrieving and rendering a small **slice** of the schema, or by letting an
agent **introspect** it through tools. Many techniques claim to do this well. This suite measures
them head-to-head on a fixed 816-query corpus across 5 schemas, with deterministic, cached scoring.

It works by **isolating one variable at a time.** Each benchmark holds everything else fixed and
varies a single axis, so a result is attributable to that axis alone:

| `pnpm eval <x>`          | what it varies                        | what it measures           | family                 | detail                                                                               |
| ------------------------ | ------------------------------------- | -------------------------- | ---------------------- | ------------------------------------------------------------------------------------ |
| **models**               | the **embedding model**               | retrieval recall@K         | semantic search        | [`src/benchmarks/models/README.md`](src/benchmarks/models/README.md)                 |
| **templates**            | the text embedded **per field**       | field recall@K             | semantic search        | [`src/benchmarks/templates/README.md`](src/benchmarks/templates/README.md)           |
| **type-templates**       | the text embedded **per type**        | type recall@K              | semantic search        | [`src/benchmarks/type-templates/README.md`](src/benchmarks/type-templates/README.md) |
| **strategies** (default) | the **selection / slicing algorithm** | **perfect %** + token cost | semantic search        | [`docs/strategies.md`](docs/strategies.md)                                           |
| **agent**                | **chat model × strategy × prompt**    | **success %** + cost       | semantic introspection | [`docs/agent-benchmark-design.md`](docs/agent-benchmark-design.md)                   |

The first four are the **semantic-search stack** — pure, deterministic, offline retrieval scored
against a known-good field set. The fifth is **semantic introspection** — a chat model discovers the
schema live through tools and has to answer a real question, end to end. Together they cover the
whole pipeline from "embed the schema" to "an agent answered the user."

The independently-verified record of everything tried — what worked, what didn't, and why — lives in
**[`insights/`](insights/README.md)**. Read it before changing a strategy, the slicer, the scoring,
or the embedding templates.

## How the semantic-search benchmarks work

`models`, `templates`, `type-templates`, and `strategies` all share one offline pipeline. The first
three score raw **retrieval** (is the target field/type near the query in embedding space?); the
strategy benchmark scores the **selection algorithm** built on top of that retrieval, on two axes
that pull against each other:

| axis           | meaning                                                                         | want |
| -------------- | ------------------------------------------------------------------------------- | ---- |
| **perfect %**  | share of queries where _every_ must-have field survived into the slice (recall) | high |
| **token cost** | size of the rendered slice, measured with `gpt-tokenizer`                       | low  |

```
schema SDL
   │  embedded once, cached on disk (.embed-cache/)
   ▼
snapshot ──── 3 vector spaces: fields · types · sub-elements (args / input fields / enum values)
   │
   ▼   for each (query × strategy), in a worker pool
strategy.run({ snapshot, query + embedding, config })  ──▶  selectedCoords: ["Type.field", …]
   │
   ▼
buildSlice() ── renders the selected coords into valid SDL, walks the type closure, and
   │            (by default) relevance-prunes optional args / input fields / enum values
   ▼
grade ──── recall: is every `mustInclude` coord present in the *rendered slice*?
           tokens: gpt-tokenizer length of the slice
   │
   ▼
runs/current/<benchmark>/results.{json,md}     (each row cached in .result-cache/ → re-runs are instant & offline)
```

Two things worth knowing:

- **Recall is graded on the rendered slice, not the raw coord list.** When a strategy drops an
  argument, it also sheds the input-object and enum types that argument dragged in — and that
  closure effect is where most of the token savings live.
- **Strategies are pure and self-contained.** A strategy is one function,
  `run(input) => { selectedCoords }`, that may import **only** [`src/core/types.ts`](src/core/types.ts).
  It receives a precomputed snapshot (field/type/element embeddings + cosine-to-query helpers) and the
  embedded query; it never calls the network or parses SDL itself.

## How the agent (semantic-introspection) benchmark works

The agent benchmark is the realistic counterpart: instead of grading a slice against a coordinate
list, it hands a chat model the same NL question a user would ask and makes it _work_. The model
never sees the schema — it discovers it through tools, one slice at a time, and must produce a
correct answer.

```
user question (NL)
   │
   ▼
chat model  ◄──────────────────────────────────────────────┐  bounded tool-use loop
   │  picks ONE tool per turn                                │  (maxTurns / maxToolCalls / maxCost)
   ├─ search(searchQuery) ─────► embed → slicer → SDL slice ─┤  slices accumulate over the union
   ├─ execute_graphql_operation(query) ─► run against the ───┤  mock returns exactly what a real
   │                                       MOCK GraphQL API   │  API would (no coverage/oracle leak)
   └─ answer({...}) ───────────► structured answer ──────────┘
   │
   ▼
grade ──── deterministic, tolerant deep-equal of the submitted `answer` against the gold
           `answer`/`answers` value in the query YAML — NO LLM judge, gold operation never shown
   │
   ▼
runs/current/agent/results.md     (success% + Wilson CI + cost/turn/failure taxonomy; rows cached)
```

What makes it honest and reproducible:

- **Three tools, one per turn.** `search` embeds a natural-language description, runs the configured
  slicer strategy, and returns a compact SDL slice — _accumulating_ over the union of everything
  searched so far. `execute_graphql_operation` runs the model's query against a **mock GraphQL
  server** for that schema and returns the data. `answer` submits a structured result matching the
  query's `answerSchema`.
- **The mock is a real server, not an oracle.** Each schema has hand-written resolvers
  (`src/benchmarks/agent/mock/resolvers/`) forming a natural object graph with shared references.
  `execute` returns **only what a real GraphQL API would** — never coverage, must-lists, or hints.
- **Grading is deterministic.** Success = the submitted `answer` tolerantly matches the gold answer
  (shape/key/nesting-insensitive, pagination-tolerant). No LLM judge, no derivation from the gold
  `operation` — if the mock returns wrong data you fix the mock; if more than one answer is valid you
  add an `answers` alternative.
- **Cost is bounded and cached.** Per-session caps (`--max-turns 12`, `--max-tool-calls 20`,
  `--max-cost 0.50`), prompt caching on the stable system+tools prefix, and a content-addressed
  result cache mean a completed sweep re-runs for free. The report is **cache-as-ledger**: it always
  shows the full accumulated board read from the cache; CLI filters scope only what runs _live_ this
  invocation, so running a subset never erases prior models from the board.

Chat models live in `src/agent-models/` (`claude-sonnet-4-6`, `claude-haiku-4-5`, `gpt-5`,
`gpt-5-nano`, `gpt-4-1-mini`); prompts in `src/agent-prompts/`. The embedding setup inside `search`
is held fixed (it is not the thing under test). See
**[`docs/agent-benchmark-design.md`](docs/agent-benchmark-design.md)** for the full spec and
**[`docs/agent-mock-server-plan.md`](docs/agent-mock-server-plan.md)** for the mock.

## Running it

```bash
pnpm install
export OPENAI_API_KEY=sk-...        # warms the embedding cache (the search benchmarks run fully offline afterward)

# --- semantic-search benchmarks (no chat-model cost) ---
pnpm eval                                 # the strategies benchmark (default)
pnpm eval strategies --strategy slicer    # just one strategy
pnpm eval models                          # embedding-model benchmark (also: templates, type-templates)
pnpm eval --list                          # list strategies / schemas / queries / models / agent models / prompts
pnpm eval --schema github -j 8            # filter by schema, 8 worker threads
```

```bash
# --- agent benchmark (spends real money on chat-model API calls) ---
export ANTHROPIC_API_KEY=sk-...           # for claude-* agent models
export OPENAI_API_KEY=sk-...              # for gpt-* agent models AND the search embeddings

# ALWAYS smoke-test before a sweep — cost is real.
pnpm eval agent --model gpt-4-1-mini --query gh-ext-007 --limit 1

pnpm eval agent                           # default: all loaded chat models × slicer × default prompt,
                                          #          over the cherry-pick manifest (base-only, no -kw)
pnpm eval agent --full-bench              # run every gradable query instead of the cherry-pick
pnpm eval agent --prune-cache             # delete orphaned cache entries the current determinants never read
```

Results are written to `runs/current/<benchmark>/results.{json,md}`. Run `pnpm eval --help` for the
full flag list (the agent-only flags — `--max-turns`, `--max-tool-calls`, `--max-cost`, `--samples`,
`--seed`, `--temperature`, `--limit`, `--keywords`, `--prune-cache` — are ignored by the other four).

## The corpus

816 queries across **5 schemas** — github, gitlab, linear, shopify, singapore — in
`src/queries/all-schemas/`. Each query is a YAML file that carries the ground truth for _both_
families of benchmark:

```yaml
id: shop-067
schemaId: shopify
query: 'the free shipping discount codes: title, status, and times used'
mustInclude: # ← semantic-search recall: each must survive into the rendered slice
    - QueryRoot.discountNodes
    - DiscountNode.discount
    - DiscountCodeFreeShipping.title
mustExclude: # coords that should NOT appear (over-selection signal; not a hard gate)
    - Mutation
operation: 'query { ... }' # reference document — provenance + offline sanity check; NEVER shown to the model
answer: # ← agent success: the gold value, graded by tolerant deep-equal
    discounts:
        - { title: 'Complimentary Shipping Offer', status: ACTIVE, usageCount: 381 }
answerSchema: { type: object, ... } # JSON Schema the model's `answer` tool must satisfy
```

- `mustInclude` (`Type.field`, an arg form `Type.field(arg:)`, or a bare type name) drives
  **slicing recall**. Six bare-union musts are structurally unsatisfiable by any field-level strategy
  and are carved out of the agent denominator.
- `answer` / `answerSchema` (present on **410 of 816** queries) drive **agent success**; the mock
  server's data is the source of truth for these values.
- Most queries also have a terse keyword-style `-kw` twin (same musts, different phrasing) to test
  embedding sensitivity to wording. The agent benchmark excludes `-kw` twins by default
  (opt in with `--keywords`).

## Extending it

- **Add an embedding model / field template / type template.** Drop a folder under `src/models/`,
  `src/templates/`, or `src/type-templates/` and run the matching benchmark. (These currently score
  on a single probe query — broadening the corpus is open work; see the caveat above.)
- **Add a slicing strategy.** `mkdir src/strategies/<id>` with a `meta.json`
  (`{ id, name, description, defaultConfig }`, where `id` equals the folder name) and an `index.ts`
  exporting `run(input): StrategyResult`. Import only from `../../core/types.ts`. Fork
  `ptv-lex-weighted` (clean helpers) or `slicer` (the production path), then
  `pnpm eval strategies --strategy <id>`.
- **Add an agent chat model.** `mkdir src/agent-models/<id>` with a `meta.json` (`provider`,
  `modelName`, pricing — Anthropic and OpenAI are supported), then
  `pnpm eval agent --model <id> --limit 1` to smoke-test.
- **Make a query agent-gradable.** Add an `answer` (and `answerSchema`) to its YAML and make sure the
  schema's mock server (`src/benchmarks/agent/mock/resolvers/<schema>.ts`) returns matching data.
